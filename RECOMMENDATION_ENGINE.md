## 2. Recommendation Engines & Personalization Logic

### 2.1 Next Concept Recommendation (Gaps & Weakness Resolution)
* **Purpose**: Recommend the immediate next concept node the student should focus on to resolve learning gaps or build upon existing knowledge.
* **Input Features**: Bayesian Knowledge Tracing (BKT) probability scores, Spaced Repetition (SRS) recall schedules, and concept difficulty indexes.
* **Selection Logic**:
  - Filter out concepts with a Mastery Probability $P(L_n) \ge 0.85$.
  - Identify prerequisite concept nodes for the current topic with $P(L_n) < 0.60$ and prioritize them to fill foundational gaps.
  - Sort remaining candidate concepts using a composite weight:
  
$$\text{Priority Score} = w_1 \cdot (1 - P(L_n)) + w_2 \cdot \text{RecencyWeight} + w_3 \cdot \text{PrerequisiteDepth}$$

Where weights are tuned dynamically ($w_1 = 0.5$, $w_2 = 0.2$, $w_3 = 0.3$).

## 3. Adaptive Learning Path Generator (DAG Topological Sort)

Personalized learning paths are generated by evaluating the study target as a **Directed Acyclic Graph (DAG)** of concept dependencies.

### 3.1 Prerequisite Traversal Algorithm (Topological Sort)
To generate a valid sequence of study topics, the engine performs a depth-first search (DFS) topological sort over the concept node network:

```kotlin
class LearningPathGenerator {
    private val visited = mutableSetOf<String>()
    private val tempMark = mutableSetOf<String>()
    private val orderedPath = mutableListOf<ConceptNode>()

    fun generatePath(targetConceptId: String, graph: Map<String, List<String>>, nodes: Map<String, ConceptNode>): List<ConceptNode> {
        visited.clear()
        tempMark.clear()
        orderedPath.clear()
        
        visit(targetConceptId, graph, nodes)
        return orderedPath
    }

    private fun visit(nodeId: String, graph: Map<String, List<String>>, nodes: Map<String, ConceptNode>) {
        if (nodeId in tempMark) throw IllegalStateException("Dependency cycle detected at node: $nodeId")
        if (nodeId !in visited) {
            tempMark.add(nodeId)
            val prerequisites = graph[nodeId] ?: emptyList()
            for (prereqId in prerequisites) {
                visit(prereqId, graph, nodes)
            }
            tempMark.remove(nodeId)
            visited.add(nodeId)
            nodes[nodeId]?.let { orderedPath.add(it) }
        }
    }
}
```

## 4. Vector Database & Hybrid Filtering Strategy

The system utilizes content similarity to map connections across books and recommend complementary reading.

### 4.1 Hybrid Recommendation Scoring Model
To determine which supplementary chapters to recommend, the system computes a hybrid similarity score:

$$\text{Hybrid Score} = \alpha \cdot \text{CosineSimilarity} + (1 - \alpha) \cdot \text{CollaborativeOverlap}$$

* **Cosine Similarity**: Measures semantic overlap between chapter text embeddings ($v_{\text{user}}$ and $v_{\text{chapter}}$) computed using the Google Vertex AI Embeddings API.
* **Collaborative Overlap**: Calculated from the completion rates of students with similar career or academic goals.
* **Hyperparameter $\alpha$**: Dynamically adjusted (default: $0.6$) to balance semantic alignment with user community trends.

## 5. Explainable AI & Prompt Library

Every recommendation is accompanied by an educational explanation detailing **why** the student should study the suggested topic.

### 5.1 Recommendation Explainer Agent
* **Purpose**: Generate personalized explanations for recommended topics, framing them in the context of the student's career or study goals.
* **Model**: `gemini-3.5-flash` (Fast, highly context-aware semantic synthesis)
* **Inputs**: Target Concept Metadata, Current Student Mastery Profile, User Goals.
* **System Prompt**:
```text
You are a Personal Academic Mentor. Your task is to explain clearly and concisely why the recommended concept is the logical next step in the student's learning path.
Relate the recommended topic directly to the student's career or academic goals.
Do not exceed three sentences, and keep your tone encouraging, objective, and clear.
```
* **User Prompt**:
```text
Recommended Concept: ${conceptName}
Associated Book: ${bookTitle}
Reason for Recommendation: ${triggerReason} (e.g. "Prerequisite constraint missing", "Upcoming exam focus area")
Student Career Goal: ${careerGoal}
```
* **Structured JSON Output Schema**:
```json
{

  "properties": {
    "recommendedConceptId": { "type": "STRING" },
```

### 6.2 Cold Start Resolution Protocol
When a new student joins the platform with no historical study metrics:
1. **Interactive onboarding survey**: Captures primary learning goals (e.g., preparing for exams, professional upskilling) and target study depth.
2. **Standard Curriculums**: Onboarding surveys assign users to curated starter pathways based on their profile.
3. **First-Quiz Calibration**: Within the first $5$ minutes of study, the system serves a brief, multi-difficulty assessment. This diagnostic quiz calibrates the Bayesian Knowledge Tracing (BKT) engine and establishes the student's initial mastery baseline.

## 7. Performance, Cost & Privacy Optimization Profiles

- [x] **Client-Side Recommendation Processing**: Graph traversal and prerequisite sorting are performed locally in the Room database, saving server execution costs.
- [x] **Aggressive Metadata Denormalization**: High-frequency node parameters (titles, categories, difficulties) are stored directly inside parent documents, reducing database read operations.
- [x] **Anonymized Signal Tracking**: User telemetry metrics are indexed using randomized user IDs (`usr_xxx`). The recommendation engine does not access or store personal identifying details.
- [x] **Cached Vector Retrievals**: Cross-book similarity vectors are pre-computed daily and cached in memory, preventing redundant vector search operations during user sessions.
